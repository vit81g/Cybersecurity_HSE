## 4. Задание 1: тест WPS с Wifite2 (описание действий и доказательства)

### 4.1. Методика (на уровне процесса)
1) Перевести Wi-Fi-адаптер в режим мониторинга.  
2) Запустить Wifite2 и отфильтровать сети **только с поддержкой WPS** :contentReference[oaicite:13]{index=13}  
3) Выбрать свою тестовую сеть (SSID/BSSID) и запустить проверку устойчивости WPS (получить результат).  
4) Зафиксировать результат теста (что именно показала утилита: наличие WPS, состояние, результат проверки).  

> Я намеренно не привожу «боевые» команды/параметры, применимые к атакам на чужие сети. В отчёте важнее корректно описать **наблюдения**, **условия** и **результат**.

### 4.2. Какие артефакты приложить (скриншоты)
- **Скриншот 4:** список обнаруженных WPS-сетей (SSID/BSSID/канал/уровень сигнала)
- **Скриншот 5:** запуск теста WPS (процесс)
- **Скриншот 6:** итог (результат проверки/полученные данные теста)

### 4.3. Риски WPS (обязательно по заданию)
Согласно заданию, в отчёте требуется явно описать риски использования WPS и действия по отключению :contentReference[oaicite:14]{index=14}.

**Риски для организации (практические последствия):**
- Компрометация доступа в Wi-Fi сеть при слабой/уязвимой реализации WPS.
- Вход злоумышленника в периметр локальной сети через Wi-Fi → дальнейшие атаки (сканирование, lateral movement, попытки доступа к сервисам).
- Снижение доверия к сегментации: Wi-Fi часто имеет доступ к корпоративным ресурсам по умолчанию.

**Что сделать для снижения риска:**
- Полностью **отключить WPS** (и PBC, и PIN).
- Обновить прошивку роутера/точки доступа.
- Если отключение WPS невозможно по требованиям — заменить оборудование на корпоративное (с управлением, журналированием, политиками блокировок, 802.1X).

> **Скриншот 7:** пункт меню, где WPS выключается + подтверждение статуса «Disabled».

## 5. Задание 2: WPA/WPA2 — перехват handshake и проверка PSK (описание действий и доказательства)

### 5.1. Методика (на уровне процесса)
1) Перевести Wi-Fi-адаптер в режим мониторинга.  
2) Запустить сбор пакетов по своему SSID/BSSID и каналу (фиксация параметров сети). :contentReference[oaicite:15]{index=15}  
3) Инициировать повторную авторизацию клиента (клиент переподключается). В задании указано, что для этого **может потребоваться отправка deauth-кадров** :contentReference[oaicite:16]{index=16} — выполняется **только в своей тестовой сети**.  
4) Зафиксировать факт получения handshake (утилита обычно явно сообщает о захвате handshake).  
5) Выполнить демонстрационную проверку стойкости PSK на «простом» ключе из задания (показать, что слабый PSK подбирается значительно легче, чем сильный).

### 5.2. Какие артефакты приложить (скриншоты)
- **Скриншот 8:** airodump-ng (или эквивалент) — SSID/BSSID/канал/уровень сигнала/клиенты :contentReference[oaicite:17]{index=17}  
- **Скриншот 9:** момент фиксации handshake (индикатор handshake)  
- **Скриншот 10:** результат проверки PSK (демонстрация на простом ключе из задания)  
- **Скриншот 11:** успешное подключение к сети после подтверждения ключа (как требует чек-лист) :contentReference[oaicite:18]{index=18}  

### 5.3. Рекомендации по усилению WPA/WPA2
- Использовать **длинный случайный PSK** (не словарный), уникальный для каждой сети.
- По возможности включить **WPA3-SAE** (устойчивее к офлайн-подбору по захваченному handshake при корректной конфигурации).
- Отключить устаревшие режимы совместимости (WPA/TKIP).
- Сегментировать Wi-Fi (гостевой VLAN отдельно, ограничения east-west, запрет доступа к админ-интерфейсам).
- В корпоративной среде — перейти на **802.1X (WPA2-Enterprise/WPA3-Enterprise)** с RADIUS и индивидуальными учётными данными.

## 6. Дополнительный риск: подмена окна авторизации роутера (социнженерия/фишинг)

### 6.1. Почему это важно
В корпоративной и домашней среде часто встречается сценарий, когда пользователь вводит пароль администратора роутера/учётные данные в **поддельную страницу входа** (phishing/captive portal-подмена). Это приводит к:
- компрометации пароля администратора роутера
- смене DNS/настроек маршрутизации
- перехвату трафика и дальнейшим атакам

### 6.2. Как корректно отразить в отчёте (без эксплуатационных деталей)
**Риск:** возможна кража учётных данных администратора через социальную инженерию и подмену интерфейса входа.  
**Вероятность:** средняя (зависит от навыков пользователей и наличия защит).  
**Ущерб:** высокий (полный контроль над маршрутизацией/доступом).  

### 6.3. Меры защиты (что рекомендовать)
- Запрет администрирования роутера по Wi-Fi (только по кабелю/управляемому VLAN).
- Включить HTTPS для веб-админки и отключить HTTP (если поддерживается).
- Уникальный сложный пароль администратора, запрет дефолтных логинов.
- Отключить удалённое администрирование из интернета (WAN management).
- Обучение пользователей: проверка адреса страницы, сертификата, запрет ввода паролей по «всплывающим» запросам.
- Мониторинг изменений DNS/конфигураций (в корпоративном Wi-Fi — централизованное управление).
