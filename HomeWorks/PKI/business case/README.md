# NWTraders — Проектирование инфраструктуры открытых ключей (PKI)

## О проекте

Данный репозиторий содержит переработанное решение учебного бизнес-кейса **NWTraders**.  
В новой версии основное техническое решение построено не вокруг общей сетевой архитектуры, а вокруг **инфраструктуры открытых ключей (PKI)**, так как именно PKI позволяет реализовать:

- строгую аутентификацию пользователей и администраторов;
- конфиденциальность и целостность данных при передаче;
- безопасный доступ к корпоративным сервисам;
- защищённое взаимодействие сотрудников, клиентов и партнёров;
- централизованное управление сертификатами, их выпуском, продлением и отзывом.

---

## Почему README был переработан

Предыдущая версия решения делала акцент на:

- сегментации сети;

Однако для данного кейса этого недостаточно.  
Центральной частью решения должна быть именно **PKI**, а инфраструктурные меры должны рассматриваться как среда, в которой PKI развёрнута и используется.

Поэтому в обновлённой версии README основной акцент сделан на:

- архитектуре удостоверяющих центров;
- шаблонах сертификатов;
- строгой аутентификации через сертификаты;
- конфиденциальности через TLS, IPsec/VPN и S/MIME;
- ролях PKI;
- жизненном цикле сертификатов;
- узлах PKI и их сетевых взаимодействиях.

## Цель работы

Разработать **технический проект PKI** для распределённой инфраструктуры NWTraders, обеспечивающий:

- строгую аутентификацию пользователей;
- безопасную аутентификацию администраторов;

- безопасное взаимодействие сотрудников по почте и web-сервисам;

## Ключевые задачи PKI в кейсе

PKI в данной работе должна решать следующие прикладные задачи:

1. Выпуск сертификатов для пользователей, устройств, серверов и сетевых сервисов.
2. Реализация строгой аутентификации на основе сертификатов.
3. Защита каналов связи между площадками и пользователями.
4. Шифрование и подпись почтовых сообщений.
5. Проверка статуса сертификатов через CRL и OCSP.
6. Управление жизненным циклом сертификатов.
7. Разделение ролей и полномочий в рамках администрирования PKI.

---

## Общая архитектура PKI

Для NWTraders предлагается **двухуровневая иерархия PKI**:

### Уровень 1 — Root CA

- **Standalone Root CA**
- размещается в Москве;
- **не входит в домен**;
- работает в **офлайн-режиме**;
- используется только для:
  - подписи сертификатов подчинённых УЦ;
  - публикации корневого сертификата;
  - публикации CRL Root CA.

### Уровень 2 — Issuing CA

- **Enterprise Issuing CA (Москва)**
- **Enterprise Issuing CA (Санкт-Петербург)**

Именно выпускающие УЦ обеспечивают:

- выпуск сертификатов пользователям;
- выпуск сертификатов компьютерам и серверам;
- выпуск сертификатов для NPS/RRAS/VPN/Wi‑Fi;
- выпуск сертификатов для web-сервисов;
- выпуск сертификатов S/MIME;
- автоматическое продление и отзыв сертификатов.

```text
             +--------------------+        +--------------------+
             |   ISS-CA-MSK       |        |   ISS-CA-SPB       |

                    |       |                      |       |
                    |       |                      |       |
            +-------+       +------+       +------+       +-------+
            |                      |       |                      |
      +-----------+         +-----------+ +-----------+     +-----------+
      | PKI-PUB   |         |  OCSP     | |  CEP/CES  |     |  SIEM     |
      | CRL / AIA |         | Responder | | Enrollment|     | Monitoring|
      +-----------+         +-----------+ +-----------+     +-----------+
            |
    -------------------------------------------------------------
    |              |                |                |          |
 Users         Devices          VPN/NPS         Wi‑Fi/NPS   WEB/OWA
```

## Узлы PKI и их роли

| Узел | Роль | Назначение |
|---|---|---|
| **ROOT-CA-MSK** | Root CA | Корень доверия, подпись подчинённых УЦ |
| **ISS-CA-MSK** | Enterprise Issuing CA | Выпуск сертификатов в Москве |
| **ISS-CA-SPB** | Enterprise Issuing CA | Выпуск сертификатов в Санкт-Петербурге |
| **PKI-PUB-MSK / SPB** | CDP / AIA | Публикация CRL и сертификатов CA |
| **OCSP-MSK / SPB** | Online Responder | Проверка статуса сертификатов |
| **CEP-MSK / SPB** | Policy Service | Выдача политики выпуска |
| **CES-MSK / SPB** | Enrollment Service | Выпуск и продление сертификатов по HTTPS |
| **PAW-PKI-ADM** | Admin workstation | Безопасное администрирование PKI |
| **SIEM** | Мониторинг | Централизованный аудит и контроль событий PKI |

## Строгая аутентификация через PKI

В новой версии решения строгая аутентификация реализуется не через общую формулировку MFA, а через **certificate-based authentication**.

### 1. Администраторы

Для администраторов Tier 0 и Tier 1 предлагается:

- использовать **Smart Card Logon**;
- выделить отдельные административные рабочие станции;
- запретить выполнение критических операций с обычных пользовательских устройств.

### 2. VPN

Для удалённого доступа сотрудников предлагается:

- **EAP-TLS**;
- пользовательские или машинные сертификаты;
- серверный сертификат на NPS/RRAS;
- запрет подключения без доверенного сертификата.

### 3. Корпоративный Wi‑Fi

Для корпоративного Wi‑Fi предлагается:

- **802.1X EAP-TLS**;
- сертификатная аутентификация пользователей и устройств;
- отказ от доступа по общему паролю сети.

### 4. Доменные сервисы

Контроллеры домена и RODC получают собственные сертификаты, что поддерживает сертификатно-ориентированные сценарии аутентификации и усиливает доверенную инфраструктуру.

---

## Конфиденциальность через PKI

Конфиденциальность в данной модели обеспечивается не разделением ролей и не RBAC, а **криптографическими механизмами PKI**.

### TLS

Используется для:

- web-портала;
- OWA;
- reverse proxy;
- внутренних web/API-сервисов;
- CEP/CES;
- OCSP.

### IPsec / VPN

Используется для:

- site-to-site VPN между площадками;
- удалённого доступа сотрудников;
- аутентификации VPN-шлюзов по сертификатам.

### S/MIME

Используется для:

- подписи электронных сообщений;
- шифрования чувствительной переписки;
- защиты коммуникаций руководства, финансового блока и ИБ.

---

## Жизненный цикл сертификатов

PKI должна обеспечивать полный жизненный цикл сертификатов:

### Выпуск

- автоматически — для рабочих станций, серверов и части пользователей;
- вручную — для администраторов, смарт-карт, критичных ролей.

### Продление

- через **auto-enrollment** для доменных объектов;
- через **CEP/CES** для удалённых и внешних устройств.

### Отзыв

Отзыв сертификата выполняется при:

- увольнении сотрудника;
- компрометации устройства;
- потере токена или смарт-карты;
- подозрении на компрометацию закрытого ключа;
- выводе сервера из эксплуатации.

### Проверка статуса

- через **CRL**;
- через **OCSP**.

## Роли и полномочия в PKI

| Роль | Ответственность |
|---|---|
| **CA Administrator** | Конфигурация и обслуживание УЦ |
| **Certificate Manager** | Одобрение и отзыв сертификатов |
| **Auditor** | Проверка журналов и контроль действий |
| **Backup Operator** | Резервное копирование и восстановление |
| **PKI Security Officer** | Контроль процедур безопасности PKI |

### Принцип разделения ролей

В PKI должно быть реализовано жёсткое разделение обязанностей:

- администратор CA не должен одновременно утверждать сертификаты;
- оператор резервного копирования не должен изменять конфигурацию УЦ;
- аудитор не должен иметь права выпуска сертификатов.

## Инфраструктурная реализация PKI

Несмотря на то что PKI является центральным решением, для её корректной работы требуется инфраструктурная среда:

- сегментация сети;
- выделенный административный сегмент;
- защищённая DMZ для опубликованных PKI-сервисов;
- изоляция гостевого Wi‑Fi;
- NGFW на периметре;
- VPN для удалённых площадок;
- журналирование и SIEM.

Таким образом:

- **PKI — это ядро доверия**;
- **сеть и защитная инфраструктура — это среда её функционирования**.

---

## Что такое RODC

**RODC (Read-Only Domain Controller)** — это контроллер домена Active Directory только для чтения, предназначенный для удалённых площадок.  
Он обеспечивает локальную аутентификацию пользователей на удалённых объектах, снижая риски по сравнению с обычным writable-контроллером домена.

---

## Ключевые запреты в проекте

В рамках проектируемой PKI должны быть зафиксированы следующие запреты:

- Root CA не должен быть постоянно подключён к сети;
- Issuing CA не должны публиковаться напрямую в Интернет;
- доступ к CA разрешён только из административного сегмента;
- внешние клиенты не должны иметь прямой доступ к AD DS и CA;
- сертификаты администраторов не должны выдаваться полностью автоматически;
- любые изменения шаблонов, ролей и конфигурации PKI должны журналироваться.

## Мониторинг и аудит PKI

В систему мониторинга должны передаваться события:

- выпуска сертификатов;
- отказов в выпуске;
- отзыва сертификатов;
- публикации CRL;
- работы OCSP;
- изменения конфигурации CA;
- изменения ролей PKI;
- действий администраторов PKI;
- ошибок CEP/CES;
- событий компрометации ключей и инцидентов безопасности.
