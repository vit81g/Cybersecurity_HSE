# NWTraders — Проектирование системы информационной безопасности

## О проекте

Данный репозиторий содержит переработанное решение учебного кейса по проектированию системы информационной безопасности для компании **NWTraders**.  
В новой версии акцент сделан не только на перечень мер защиты, но и на **техническое решение**: архитектуру, сегментацию сети, размещение узлов, схему публикации сервисов, удалённый доступ и таблицу разрешённых сетевых потоков.

---

## Исходные данные кейса

Компания **NWTraders** имеет распределённую инфраструктуру:

- **Москва** — штаб-квартира;
- **Санкт-Петербург** — второй крупный офис;
- магазины и склады;
- удалённые заготовительные точки;
- мобильные сотрудники, работающие через **3G/LTE**;
- **web-портал** для клиентов и партнёров.

### Используемая инфраструктура

- **Active Directory Domain Services**
- единый домен: `nwtraders.msft`
- контроллеры домена в Москве и Санкт-Петербурге
- **RODC** на удалённых площадках
- **Microsoft Exchange**
- **CRM / ERP / финансовые системы**
- **VPN** через Интернет
- **корпоративный и гостевой Wi‑Fi**

---

## Цель работы

Разработать **архитурно-техническое решение** по информационной безопасности для распределённой инфраструктуры компании, обеспечивающее:

- защиту критичных данных при хранении и передаче;
- безопасное взаимодействие между площадками;
- строгую аутентификацию пользователей;
- безопасный доступ мобильных сотрудников;
- безопасную публикацию внешних сервисов;
- снижение риска компрометации внутренней инфраструктуры.

---

## Ключевые активы

В рамках модели угроз к критичным активам относятся:

- база данных **CRM**;
- **ERP-система**;
- **финансовая отчётность**;
- учётные записи и группы **Active Directory**, особенно привилегированные;
- почтовая система **Exchange**;
- **web-портал** клиентов и партнёров;
- инфраструктура **VPN**;
- контроллеры домена и сервисы аутентификации.

---

## Основные угрозы

### Внешние угрозы

- фишинг и компрометация учётных данных;
- brute-force против VPN и OWA;
- атаки на web-портал;
- перехват трафика в публичных сетях;
- DDoS-атаки.

### Внутренние угрозы

- злоупотребление привилегиями;
- утечка данных сотрудниками;
- подключение несанкционированных устройств;
- распространение вредоносного ПО внутри сети.

### Технологические угрозы

- компрометация Active Directory;
- атаки на удалённые площадки и RODC;
- отсутствие сегментации сети;
- уязвимости в CRM / ERP / web-портале;
- неконтролируемый беспроводной доступ.

---

## Техническая архитектура решения

Предлагаемое решение строится по принципу **многоуровневой защиты** с разделением инфраструктуры на независимые зоны безопасности.

### Общая логика архитектуры

- на периметре каждого крупного офиса размещается **NGFW**;
- внешние сервисы публикуются только через **DMZ**;
- взаимодействие площадок осуществляется через **site-to-site VPN**;
- мобильные сотрудники подключаются через **Remote Access VPN** с **MFA**;
- внутренние серверы и доменная инфраструктура размещаются в изолированных сегментах;
- гостевой Wi‑Fi полностью отделён от корпоративной сети;
- события безопасности централизованно собираются в **SIEM**.

---

## Сегментация сети

В инфраструктуре предлагается выделить следующие зоны:

1. **Internet** — внешняя недоверенная сеть.
2. **Perimeter / NGFW** — пограничный уровень фильтрации и контроля.
3. **DMZ** — зона публикации внешних сервисов.
4. **VPN Zone** — зона удалённого защищённого доступа.
5. **User LAN** — рабочие станции пользователей.
6. **Server LAN** — внутренние серверы приложений и сервисов.
7. **Critical Infrastructure LAN** — AD DS, PKI, DNS, иные критичные сервисы.
8. **Admin LAN** — отдельный административный сегмент.
9. **Corporate Wi‑Fi** — корпоративная беспроводная сеть.
10. **Guest Wi‑Fi** — гостевая сеть только с выходом в Интернет.
11. **Remote Sites** — удалённые магазины, склады и точки заготовки.

## DMZ и публикация внешних сервисов

В зоне **DMZ** должны размещаться только сервисы, доступные извне:

- **WAF / Reverse Proxy**;
- внешний узел публикации **OWA**;
- web-узел портала;
- VPN-шлюз удалённого доступа.

### Важное ограничение

В **DMZ не размещаются**:

- контроллеры домена;
- базы данных;
- внутренние серверы Exchange;
- ERP / CRM;
- финансовые системы.

Это снижает риск прямого доступа к внутренней инфраструктуре при компрометации внешнего узла.

---

## Защита Active Directory

Так как Active Directory является ключевым элементом инфраструктуры, её защита выделяется в отдельный контур.

### Меры защиты

- выделенный сегмент для контроллеров домена;
- разделение учётных записей: пользовательские и административные;
- применение **MFA** для привилегированного доступа;
- использование **PAW** (Privileged Access Workstations);
- ограничение административного доступа только из **Admin LAN**;
- аудит входов, изменений групп, GPO и привилегий;
- передача событий в **SIEM**.

### Что такое RODC

**RODC (Read-Only Domain Controller)** — это контроллер домена Active Directory **только для чтения**, предназначенный для удалённых площадок.  
Он позволяет выполнять аутентификацию на месте, но не хранит полноценный writable-функционал обычного контроллера домена. Такое решение снижает риск компрометации доменной инфраструктуры в филиалах и на удалённых точках.

## Защита каналов связи и удалённого доступа

### Между площадками

Для связи между Москвой, Санкт-Петербургом, магазинами, складами и удалёнными точками используются:

- **site-to-site IPsec VPN**
- шифрование трафика поверх Интернета
- аутентификация узлов по сертификатам

### Для мобильных сотрудников

Для сотрудников, работающих из внешних сетей:

- **Remote Access VPN**
- обязательная **MFA**
- контроль состояния устройства
- ограничение доступа по ролям

## Защита данных

### При хранении

- полнодисковое шифрование рабочих станций и ноутбуков;
- шифрование критичных серверных томов;
- шифрование резервных копий;
- прозрачное шифрование БД при необходимости.

### При передаче

- **TLS** для web-сервисов;
- **IPsec VPN** между площадками;
- запрет устаревших протоколов и небезопасных алгоритмов.

## Таблица разрешённых сетевых потоков

Ниже приведён укрупнённый перечень разрешённых взаимодействий между зонами.

| № | Источник | Назначение | Протоколы / порты | Назначение потока | Политика |
|---|---|---|---|---|---|
| 1 | Internet | WAF / Reverse Proxy (DMZ) | TCP 443 | Доступ клиентов и партнёров к web-порталу | Разрешить |
| 2 | Internet | VPN Gateway | UDP 500, UDP 4500, ESP / TCP 443 | Подключение удалённых сотрудников и площадок | Разрешить |
| 3 | Internet | OWA Publishing Node | TCP 443 | Удалённый доступ к почте | Разрешить |
| 4 | Internet | Internal Segments | Любые | Прямой доступ во внутреннюю сеть | Запретить |
| 5 | DMZ | Web/App Server | TCP 443 / прикладной порт | Передача запросов от reverse proxy к приложению | Разрешить адресно |
| 6 | Web/App Server | DB Server | TCP 1433 / порт СУБД | Доступ приложения к базе данных | Разрешить адресно |
| 7 | User LAN | AD DS | DNS 53, Kerberos 88, LDAP/LDAPS 389/636, SMB 445 | Аутентификация и доменные сервисы | Разрешить ограниченно |
| 8 | User LAN | Exchange / Internal Apps | TCP 443 и регламентные порты | Доступ пользователей к корпоративным сервисам | Разрешить ограниченно |
| 9 | User LAN | Admin LAN | Любые | Доступ пользователей к административным хостам | Запретить |
| 10 | Admin LAN | Server / Critical LAN | RDP, WinRM, SSH и служебные протоколы | Администрирование | Разрешить только администраторам |
| 11 | Guest Wi‑Fi | Internet | HTTP/HTTPS, DNS | Гостевой доступ | Разрешить |
| 12 | Guest Wi‑Fi | Any Internal Segment | Любые | Доступ в корпоративную сеть | Запретить |
| 13 | Remote Sites (VPN) | Central Services | Только необходимые сервисы | Работа удалённых точек | Разрешить только через VPN |
| 14 | Infrastructure / Security Nodes | SIEM | Syslog, WEF, агентский трафик | Централизованный мониторинг | Разрешить адресно |

## Ключевые запреты

В архитектуре должны быть жёстко зафиксированы следующие ограничения:

- прямой доступ из Интернета к внутренним серверам;
- доступ из гостевого Wi‑Fi во внутреннюю инфраструктуру;
- прямой доступ пользователей к административному сегменту;
- неограниченный исходящий доступ серверов в Интернет;
- доступ в корпоративные сервисы в обход VPN;
- использование устаревших протоколов и слабых алгоритмов.

## Мониторинг и реагирование

Для контроля событий безопасности предлагается централизованный **SIEM**, принимающий события с:

- контроллеров домена;
- серверов Exchange;
- NGFW и VPN-шлюзов;
- WAF / reverse proxy;
- web-портала;
- CRM / ERP;
- административных рабочих станций;
- систем управления мобильными устройствами.

### Примеры сценариев мониторинга

- множественные неуспешные попытки входа;
- успешный вход после серии неудачных;
- изменение привилегированных групп;
- изменение GPO;
- нетипичный вход администратора;
- аномальная активность на VPN;
- подозрительная активность на web-портале;
- попытки доступа из гостевой сети к внутренним ресурсам.
